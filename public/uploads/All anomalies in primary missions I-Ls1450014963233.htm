E-10451 S-BBBBB-0002 07/09/2009 12:00:00   Star Tracker Power Cycle         CLOSED   FLIGHT SEGMENT THEMIS ACS-ATTITUDE CONTROL SYSTEM ACS     ACS Attitude Control Subsystem   Star Tracker                     10/19/2008 190                       263 Doe         No Effect No Effect None       Johnson   Software Environmental 01/19/2011 N N 07/24/2009 Star Tracker Power Cycle due to invalid quaternions for > 30 seconds.   The ACS software causes the star tracker to power cycle if it goes 30s without a valid quaternion. This occurred due to a bit flip in the Star Tracker RAM in a part of the code that gets accessed during outages. There was a partial outage on 7/9/09 when the power cycle occurred. The power cycle rewrites the RAM and the Star Tracker returns to full functioning. Science data was not lost or interrupted. ACE FD&C autonomously power cycles the star tracker. This corrects the problem. Since the power cycle is part of ACS FD&C, the FDC counters incremented and FDC latches were set. The following two procedures were followed to reset these telemetry points: S924-Op31210 Reset ACE FDC latch Operating procedure S924-Op31211 Reset ACS FDC totals See attached Command Action Request # 296 for the exact action taken. None.   7/9/09 - See Orbital RAZOR Issue. 10/22/09 - Star tracker power cycle without partial eclipse. See attached : CAR_325.doc 12/10/09 - Star Tracker power cycle during partial eclipse. See attached : CAR_341-cr.doc, StarTrackerPowerCycle-DTU-Response.pdf, ibex_star_tracker_FDC_power_Cycles.xls. 12/17/09 - Star tracker power cycle during partial eclipse. The CAR to clear the FDC flags has been made into a Standing CAR (executed every time the star tracker power has been cycled, dates executed added to the CAR). E-10549 S-BBBBB-0003 08/09/2009 10:12:03 2009-221-10:12:03 BBBBB Flight Computer Reboot         CLOSED   FLIGHT SEGMENT Fermi C&DH-COMMAND AND DATA HANDLING C&DH     C&DH Command & Data Handling FC Flight Computer     1514173               10/19/2008 221 40                 D S 294 Doe   Jane Doe      Minor Minor Data Loss       Johnson   Software Hardware Failure 01/19/2011 N N 08/20/2009 The BBBBB Flight Computer rebooted due to a watchdog timer error. Onboard FD&C removed power to the payload, rebooted the Flight Computer and put the spacecraft in it's safest state, Contingency State. The spacecraft and payload health was as expected after a reboot. One orbit of data (one swath of the sky) was lost due to SSR data corruption. This swath has already been imaged once in BBBBB nominal operations, so the net effect is a reduction in statistics in that swath of the sky.   The Flight Computer (FC) reset is a known problem on Orbital’s Microstar S/C bus, that we expected to happen occasionally and had procedures already in place to handle 8/10/09 The 9:35pm EDT blind acquisition pass went well. Upon contact with the spacecraft, all systems were within limits with values expected in Contingency State. After command link was verified we uploaded a Transmitter OFF command. After verifying spacecraft SOH in Contingency State the Contingency Recovery procedure was started. The procedure ran smoothly and configured the spacecraft in Housekeeping State with all telemetry nominal. The CEU was then powered on. All payload housekeeping was nominal. The SSR read & write pointers both read 0, indicating that the SSR was not in self refresh mode (see explanation at end of email). Because of this the data on the recorder is likely to be corrupted. The SSR blocks were dumped as planned in 3 stages: 1) data surrounding the time of the anomaly, 2) data for the entire orbit, 3) data since the CEU power-on. All of these dumps were executed successfully. The spacecraft is in Housekeeping State and the payload is in LVENG mode. All systems are functioning nominally. 8/11/09 The inertial precession maneuver was successful. During tonight's first pass telemetry was nominal. ATSs for the 0700-0800 contact and the inertial precession maneuver were uploaded. The maneuver occurred during the 0700 pass. The burn itself could not be monitored due to ground contact problems, but once telemetry was restored the successful completion of the maneuver was verified. 8/12/09 We successfully completed payload recovery. All payload and spacecraft bus systems are nominal and the payload is collecting data. The night's activities went very smoothly. During the first pass commands for the second pass were uploaded, BBBBB-Lo then -Hi were turned on and telemetry was verified. We entered LVSCI mode and verified the spin pulse was received and equivalent to the spin pulse the ACS system thought it was sending out. BBBBB-Lo high voltage was enabled then -Hi's HV was enabled. During the second pass both sensors entered HVSTANDBY levels and BBBBB-Lo was ramped to HVSCI levels. We entered HVSCI and dwelled between passes. The third pass was a blind acquisition. After uploading a transmitter off command for the end of the pass BBBBB-Hi was ramped to HVSCI levels and the payload entered HVSCI. Both sensors are collecting data & count rates look good.       Mary Johnson  01/19/2011 N Significant Y
